    Agentic AI 在高度受限環境下的規劃與反思機制：2024–2026 細粒度深度分析

    研究範圍與「高度受限」的可量化定義

    本報告將「高度受限（highly constrained）」視為一個多維資源/風險向量，而不是單一條件。因為在 Agentic AI（具目標導向、可多步互動、可規劃與自我修正的系統）中，同一個限制（例如「離線」）會同時影響工具使用、記憶體策略、規劃深度與反思頻率，且很難用單一門檻描述。

    可操作化的約束軸與常用量化單位

    計算/吞吐（Compute / Throughput）：常用指標是 tokens/s（輸出吞吐）與每 token 的能耗（J/token）。在能源與延遲受限時，token 的「單位成本」會直接決定可用的規劃深度與反思次數。

    記憶體（RAM/VRAM/DRAM）與 KV-cache 壓力：對自回歸 Transformer 而言，長上下文與長推理軌跡會因 KV-cache 成長而快速吃掉可用記憶體，導致 OOM 或必須做 swap/壓縮，進一步增加延遲與能耗。

    能源（Energy / Power）：以電池供電或熱設計功耗（TDP）很低的裝置上，關鍵不是「能否跑得動」，而是「每決策循環能消耗多少 J、能撐多久」。在 Raspberry Pi 4 這類 SBC 上，實測平均能耗可能達到多 J/token 等級，意味著一次多輪規劃/反思很快就會把能量預算燒光。

    延遲（Latency / Deadline）：Agentic 系統常是「感知→決策→行動」閉環，若有硬即時或近即時需求，規劃與反思必須有可預測的最壞情況上界（bounded worst-case）。在邊緣 LLM 推論的工作中，常以「每 token 的即時 DVFS 控制、或分層排程」來兼顧 deadline 與能耗。

    網路（Offline / Intermittent connectivity）：離線或間歇連線會改寫「工具可得性」與「知識更新」策略。值得注意的是：在「純設備能耗」角度，有研究顯示把生成搬到遠端伺服器（透過 HTTP 取得回答）可能比本地生成更節能，但這需要穩定連線且犧牲資料主權與可用性。

    安全/確定性（Safety / Determinism）：高度受限情境常同時伴隨高風險（醫療、工控、交通、太空）。這會把「反思」從提升成功率的技巧，提升成需要可監控、可介入、可審計的機制（例如 runtime monitor 估計進入不安全狀態的風險，超過門檻即觸發干預）。

    典型場景拆解：限制如何迫使規劃/反思改型

    以下列舉的「典型場景」在 2024–2026 的 edge LLM 文獻中反覆出現；差異在於它們同時疊加了 SWaP（Size/Weight/Power）、KV-cache 記憶體壓力、以及工具/網路限制。

    • 邊緣裝置上的自主機器人 / 無人機：常見特徵是「動態環境 + deadline + 高風險」。因此規劃通常需要分層（高層語意/任務規劃 vs 低層控制器），反思也必須可被限頻與限深度，避免累積延遲。
    • 穿戴式/行動端醫療與監測：更偏「長期在線、極低功耗」，反思與規劃往往只能是稀疏觸發（event-triggered），且很多安全關鍵判斷會繞過 LLM。
    • 離線 IoT / 感測器網路：網路不穩定使工具使用成本極高（甚至不可用），迫使 Agentic AI 走向「小模型 + 規則/狀態機 + 極簡推理 token」的混合型。
    • 太空探測器/深空任務：通信延遲與不可預測環境意味著「本地自治」與「形式化安全界線」更重要；規劃/反思需有可證明的資源上界與可回退策略。

    受限迫使調整的核心結論：在高度受限場景中，「規劃與反思」不再是無上限增加推理步數（think more）即可，而是必須設計成可度量、可裁剪、可在任何時刻繼續服務（anytime/interruptible）的策略，並把成本（tokens、工具呼叫、KV-cache）納入第一級指標。

    受限環境下的規劃機制設計：從 CoT 到樹狀搜尋與混合控制

    本節聚焦「規劃（planning）」在受限環境中的細粒度設計：每一步多產生多少 token、是否需要多次模型呼叫、是否需要額外 evaluator/critic、以及對記憶體（特別是 KV-cache）造成的放大效應。

    受限情境下，規劃方法的成本模型

    在「上線部署」視角，規劃機制通常可用下列成本分解（可直接映射到可量化指標）：

    • Token 成本：每決策循環輸出 token 數（含中間推理、工具輸出摘要、反思文本、投票/聚合的額外文本）。
    • 模型呼叫次數：多輪策略（多 agent debate、ToT 展開、多次 self-consistency）往往把 token 成本乘上「候選數」或「回合數」。
    • 評估器/裁判器成本：ToT、value-guided search、Reflexion 類方法常需要 evaluator/critic；在 edge 上部署獨立 verifier 可能不實際，因此會追求「共享 backbone、重用 KV-cache」的低成本 verifier。
    • KV-cache/上下文壓力：長推理序列不只增加 token，還會增加 cache footprint，進而觸發 swap 或迫使縮短上下文。

    這個分解之所以重要，是因為 2024–2026 的能耗研究多次指出：能耗與延遲高度受「回覆長度、生成 token 數」驅動，而不是單純 prompt 複雜度。

    CoT / ReAct / Plan-then-execute：在受限情境的可控化改造

    Chain-of-Thought（CoT）的優點是成本大致與 token 線性關係（更容易做上界控制），但在受限環境必須處理「過度 verbose」問題；因此近年的「budget forcing」或 token budget 內生化訓練，目標就是把推理壓縮到可預算的 token 範圍。

    在 edge 端推理部署的工作中，entity["organization","Qualcomm AI Research","research group, us"]提出用 LoRA adapter 啟用「reasoning mode」，並透過 budget-forced RL 將平均推理 token 降低 2.4×，同時維持任務準確度；這是一種把「規劃深度」做成可被硬性約束的方式。

    ReAct把推理與行動（例如查詢知識庫、與環境互動）交錯，能降低幻覺並改善多步任務完成率，但其成本往往由「互動步數 × 每步 token」主導；在離線或網路不穩定環境下，ReAct 的外部工具呼叫會變成瓶頸，因此需要把工具呼叫數做成硬 budget（或把工具改成本地化）。

    Plan-then-execute / Plan-and-solve 類方法在受限環境的關鍵設計不是「先產生更長的計畫」，而是把計畫結構化為可執行的有限集合（例如狀態機、層級任務圖），並允許在任何時刻「只執行下一步」；否則一次性長計畫會帶來不可預測延遲與較高 KV-cache 壓力。與此一致，對推理策略做「budget-aware」評估的研究發現：一些看似改進的策略，若不控制推理 budget，改善可能主要來自「多用了推理/多呼叫模型」而非方法本身。

    Self-consistency、Tree-of-Thoughts、Tree search：受限環境下何時值得用「搜尋」

    Self-consistency（SC）透過多次採樣與投票提升正確率，但在受限環境要面對「n 倍 token」的線性膨脹；因此它常需要「動態 n」與早停。對推理策略做「推理預算等價（equal-budget）」比較的 EMNLP 2024 工作指出：在多資料集與多模型條件下，CoT+SC 經常是非常強的 baseline；相較之下，多 agent debate 與 Reflexion 在 budget-aware 評估下常出現性能不升反降，顯示「加更多推理回合」並不必然帶來可用的收益。

    Tree-of-Thoughts（ToT）把推理視為樹狀搜尋，能在需要回溯/探索的任務上顯著提升成功率，但代價是：
    1) 需要生成多個候選 thought（token 成本大），2) 通常需要 evaluator/自評來選擇擴展節點（額外推理/甚至額外模型）。

    同一份 EMNLP 2024 budget-aware 研究提供了對 edge 很關鍵的細節：ToT 在較弱模型上可能不如 CoT+SC，而在較強模型（如 GPT-4）上 ToT 才可能在考慮 budget 後展現優勢；此外 evaluator 的品質對 ToT 影響「非線性」且非常大，random evaluator 會造成明顯性能崩落。

    受限環境下的設計結論：如果要用搜尋型規劃（ToT/MCTS/Tree search）而不是純線性 CoT，必須同時做到：
    - 將搜尋步數、每步最大輸出 token、工具呼叫數明確作為硬 budget；
    - 將評估器成本壓到「可忽略等級」（例如共享 backbone 的 verifier head），或使用更弱的 evaluator 但以成本效益維持性能；
    - 引入「budget-aware 的節點選擇」避免把有限計算平均灑在低價值分支。

    2026 的 Budget-Aware Value Tree Search（BAVT）提供了一個很接近「受限環境可用版本」的 tree-search 範式：它在設計上把 token budget 與 tool-call budget 做成三檔硬限制（例如 reasoning model：2000/4000/8000 token；tool calls：5/10/20），並引入「budget backstop」以保證在預算耗盡前產出答案。
    更關鍵的是它的 ablation 顯示：只有樹結構但缺乏價值導引與 budget-aware 節點選擇，平均表現反而會變差（在 multi-hop 搜尋空間中，無導引展開會把 budget 撕裂成碎片）。

    在 edge 上「工具」往往比 token 更貴：規劃必須內建工具節流

    在工具增強代理（tool-augmented agent）的世界裡，常見直覺是「多查幾次工具更安全」。但在受限環境，工具呼叫可能是：
    - 網路不可用（離線），或
    - 能耗/延遲更高（例如需要無線傳輸），或
    - 金錢/配額成本更高（雲端 API）。

    BAVT 的成本估算在「雲端 API 價格」設定下指出：外部 search 的成本在各 budget tier 中可佔總成本 90% 以上，凸顯「每一步工具呼叫是否帶來信息增益」必須成為規劃的核心判斷，而不是事後補救。
    即使不採用該估價模型，對 edge 而言也成立：工具呼叫通常代表不確定的延遲與能耗，因此規劃要把「工具呼叫次數」當作一級 budget，並設計可回退（例如只用本地 cache/只用已有觀測做出次佳決策）。

    受限環境下的反思與自我修正機制：低成本 verification、單次 revision 與頻率控制

    本節聚焦「反思（reflection / self-correction）」如何在受限環境落地：不是多做幾輪自我批評，而是設計「每一點額外計算帶來多少成功率提升」的可量化 ROI。

    「反思」在受限環境的三種可行形態

    共享 backbone 的輕量 verifier（latent verifier head）：
    在 edge 推理管線中，最有代表性的設計是：不額外部署一個完整 critic model，而是在同一個生成模型上加一個極小 verifier head（線性層+sigmoid），重用生成時已計算的表示與 KV-cache，避免重跑 prompt 與避免額外參數搬移。該設計被報告為「每個 stream 約等價一個額外 token 的開銷」，但能在複雜推理基準上帶來最高約 10% 的準確度提升。

    單次、具方向性的 revision（避免多輪 open-ended 自我修正）：
    2026 的 PACER提出「共識封包（consensus packet）→單回合修正」：先從多條推理軌跡（traces）萃取少量候選答案與代表性理據，形成 bounded-length 的封包，再讓模型做一次短 review。其實驗顯示：相對 naive self-refinement，Phase 2 的 token 使用量可下降約 17–28% 且維持準確度；在 HMMT 2025 上甚至達到 28/30，相較 majority vote baseline 25/30。

    budget-aware 的反思頻率控制（每 N 步、或事件觸發）：
    在受限環境，反思必須「限頻」，否則會累積延遲與能耗。近期 token-budget-aware 工作指出：LLM 的推理存在冗長性，可透過加入 token budget 或讓模型內生化 budget-awareness 來壓縮推理長度；核心價值是讓反思/推理能被納入固定預算。
    BAVT 類方法則更進一步把「剩餘預算」當成狀態，動態從探索轉向利用，並在預算接近耗盡時觸發 backstop 產出答案，避免因反思過深而「來不及回答」。

    反思不是越多越好：budget-aware 實驗揭示的「反思困境」

    受限環境最重要的警訊之一，是：某些反思/多輪策略在 budget-aware 評估下可能沒有收益甚至有害。

    EMNLP 2024 的 budget-aware 評估顯示：在相同推理 budget 下，CoT+SC 經常優於多 agent debate（MAD）與 Reflexion；且論文明確指出 MAD 與 Reflexion 常在增加 budget 後反而降低效能。
    此外該研究也顯示 Reflexion 若有「oracle evaluator」會顯著更好，但實務 evaluator（即便是 GPT-4 evaluator）可能仍落後於 self-consistency 版本，凸顯反思品質取決於評估器可靠度，而評估器在 edge 上往往是最難負擔的部分。

    從「整個 agent 系統成本」視角，TMLR 2025 的 AI Agents That Matter指出：在 HumanEval 等任務上，不同 agent 設計的成本差異可接近兩個數量級；其中 Reflexion 與 LDB 的成本比 warming 策略高出 50% 以上，而 LATS 的成本甚至高出 50 倍以上，顯示以「複雜反思/搜尋」堆疊出來的性能提升，若沒有成本報表，對受限部署幾乎不可用。

    綜合結論：受限環境下可行的反思應更像「低成本驗證/修補」而不是「多輪深度自我對話」。PACER 與 edge-verifier head 的共同點是：把反思成本設計成可預測、可上界，並讓每一次反思都具有「方向性信息」（共識封包或 verifier 分數），避免 open-ended 消耗。

    把反思成本量化成「每決策循環的 J 與秒」

    受限裝置上，「反思頻率」最終要回到能耗/延遲預算。以 Raspberry Pi 4 的硬體量測研究為例：模型家族的平均能耗可達 2.61±1.49 J/token（qwen2.5 0.5B）到 9.35±3.30 J/token（gemma2 2B）等級；在某些任務（如 GSM8K）下，量化後能耗差異甚至可達 8.4×（3.02 J/token 到 25.44 J/token）。
    在這種量級下，若一次反思多產生 200–500 token，就可能額外增加數百到數千 J 的能耗（且通常帶來數十秒以上延遲），因此反思必須「少而精」，並盡量採用「共享 KV-cache 的 verifier」或「單次 revision」類設計。

    硬體—演算法共同設計：從行動端 NPU 到存算一體與神經形態，及離線/混合連線策略

    本節聚焦「規劃/反思機制」如何與硬體特性共同設計：因為在 edge 上，瓶頸往往不是純算力，而是記憶體頻寬、KV-cache、以及能耗/熱限制。

    行動端/邊緣推理的關鍵結構：prefill vs decode 的異質性

    多份 2025–2026 的能耗與系統研究指出 LLM 推理存在結構性非對稱：prefill 更偏計算密集、decode 更偏記憶體/頻寬受限；因此「同樣的量化/加速技巧」在不同階段可能效果相反。
    例如 2026 的能耗研究指出：在 memory-bound 的 decode 階段，低精度格式可能因額外 dequant kernel、碎片化與 idle power 等因素，造成能耗不降反升（甚至 int8 比 float32 更耗能）。
    但在 Raspberry Pi 4 的 edge 實測中，量化通常能顯著降低能耗，最高可達 79% 下降；差異的主要原因是硬體與 runtime 堆疊不同（GPU kernel 與 batching/idle 行為 vs SBC CPU-only 推理），以及工作負載（任務、輸出長度）不同。

    以軟硬共設把「規劃/反思」做成可部署：CLONE 與 on-device reasoning pipeline

    USENIX ATC 2025 的 CLONE把「模型層（剪枝/LoRA）+ 系統層（DVFS、線上控制）」與硬體加速器共同設計，用於滿足 edge 的 SWaP 與即時性需求；其報告顯示可將推理流程加速最高 11.92×、節能最高 7.36×。

    存算一體與類比 attention：對「長上下文規劃」的潛在突破與現實限制

    長上下文對規劃/反思很重要（需要保存過去狀態、工具結果、錯誤原因），但也造成 KV-cache 與記憶體搬移成本。對此，2024–2025 的類比存算一體 attention研究提出極端能效的可能性：用 gain cell memories 讓 KV-cache 寫入與 dot-product 在記憶體內完成，報告指出 attention 計算可相對 GPU 降低最高 2 個數量級延遲與 5 個數量級能耗，並透過映射/初始化方法使 NLP 表現接近 GPT-2 等級。

    神經形態（Neuromorphic）對「事件觸發規劃/反思」的啟示

    神經形態方向的核心賣點，是事件驅動與低精度計算，理論上非常契合「少量、稀疏觸發」的規劃/反思。2024–2026 有幾條值得注意的路徑：

    • Loihi 2 上的 MatMul-free LLM：在 OpenReview 2025 的工作中，作者報告一個 370M 參數的 MatMul-free 模型可量化而無精度損失，並在初步結果中相對 edge GPU 的 transformer LLM 達到最高約 3× 吞吐與 2× 低能耗（並宣稱更好的 scaling）。
    • Sorbet（spiking transformer LM）：用硬體友善的 PTsoftmax 與 BSPN（以 bit-shifting 取代 softmax/LN 的昂貴操作）提升 neuromorphic 可部署性；但其自身也承認缺乏實體晶片測量，現階段多為 Lava/Verilog 估計，可靠性屬「證明可行、但量測仍不足」。
    • Spike-driven LLM（SDLLM）：OpenReview 2025/2026 的工作宣稱相較先前 spike-based LLM，能耗降 7.8× 且常識推理準確度提升 4.2%。這類結果若能在實體晶片上被大規模復現，將對「極低功耗 agent」非常關鍵，但目前不同 papers 間的硬體/量測方法差異很大，需要謹慎對比。

    離線 vs 混合連線：知識更新與反思策略的根本差異

    在混合連線（偶爾可上網）的情境裡，常見策略是把「重規劃/深反思」搬到雲端，把「即時決策」留在本地；但能耗研究顯示，單就客戶端能耗，本地生成可能比遠端取得回答更耗能（平均多 696.6%）；這逼迫系統設計在「能源 vs 主權/可用性」間取捨。
    因此在高度受限環境，較可行的模式往往是：
    - 離線時：採用「小模型 + 短 token + 本地工具/cache + 嚴格 budget」與可回退的控制器；
    - 有連線窗口時：批次更新知識/策略（例如更新壓縮後的檢索索引、更新 verifier 閾值、或重新蒸餾更短的推理風格），而不是線上逐步反思。

    受限環境下可用的公開量化證據與基準

    SBC/edge 端的推論能力上界：對 3 款 SBC（含 Raspberry Pi 4/5 等）與 25 個量化 LLM 的系統性測試指出：SBC 可較可靠支援到約 1.5B 參數級模型；且 inference runtime 的差異很大（Llamafile 相對 Ollama 可達 4× 吞吐、並降低約 30–40% 功耗）。這對 agentic 規劃的含義是：在純 SBC 上，想做 ToT/MAD 等多分支推理往往會直接超出功耗與時間預算。

    邊緣端能耗-準確度 Pareto：在 Raspberry Pi 4 上對 28 個預量化模型做硬體級能耗量測的研究提供了更細粒度結論：
    - 不同 base model 家族平均能耗可達 2.61±1.49 到 9.35±3.30 J/token；
    - 量化可在某些模型上帶來最高約 79% 的能耗下降，但呈現非線性（更激進量化不一定更省）；
    - 任務差異巨大：HumanEval 能耗可低至 1.94 J/token，而 BigBenchHard 可高達 11.92 J/token（約 6× 差距），與輸出長度/固定開銷攤提高度相關。

    推理/反思策略的 budget-aware 對照：EMNLP 2024 的 budget-aware 評估與 TMLR 2025 的 agent 成本研究共同提供一個關鍵訊息：在公開 demo/論文中常被視為「更強」的多輪/反思/多 agent 方法，若不做等預算對照，容易把收益誤判為方法本身；而在受限部署中，這種誤判會直接導致不可用的成本爆炸。

    能耗量測與 benchmarking 工具鏈：若要在 2025–2026 做可重現的能耗/成本比較，至少有兩個「基準化」方向值得採用：
    - TokenPowerBench：提供 Joules/token、Joules/response、瞬時功率等指標，並拆分 prefill/decode；其結果顯示 context 變長會使能耗/token 上升（例如 Llama3-70B 從 2K 到 10K prompt 可能上升約 3×），batch 變大可降低約 25%/token 等級的能耗，且 MoE 稀疏路由可把 token 能耗降 2–3×。
    - ML.ENERGY Benchmark：強調在接近真實服務情境下量測，並報告自動化建議有時可帶來 >40% 能耗節省（在不改變模型計算內容的前提下）。

    高度受限下最常見、最難解的瓶頸

    KV-cache 造成的記憶體溢位與延遲失控：在行動端推理部署工作中，作者明確指出長推理軌跡與長上下文會快速耗盡 KV-cache 記憶體，並且在切換模式/模型時重編碼 prompt 的代價在 edge 上極高；這直接限制了「長期規劃」與「多輪反思」的可行性。

    反思導致的累積延遲與收益遞減：多輪反思/多 agent debate 可能引發「錯誤傳播、答案多樣性下降」等效應，甚至在更大 budget 下表現更差；這在 budget-aware 實驗中已被觀察到。

    極端量化的不穩定與可預測性下降：Raspberry Pi 實測指出某些高度量化變體的能耗變異可非常大（標準差可接近均值），代表在需要 deterministic latency/energy 的情境（例如硬即時）會很難做保證。

    「量化一定更省電」的矛盾現象：在 GPU 服務情境的研究中，int8/int4 在 decode 階段可能因 dequant overhead、kernel fragmentation、以及高 idle power 而不省反耗；而在 SBC 上量化通常省能耗。兩者矛盾的可能原因包括：硬體（GPU vs CPU/NPU）、工作負載（batching/arrival shaping）、以及 runtime 實作（量化是否真正減少記憶體搬移）。

    安全與可控性：在高風險情境，「反思」需要被外部監管而不是完全自我主導。2026 的 ProbGuard 用 DTMC/風險預測做 proactive runtime monitoring，提供一種把 agent 行為抽象成狀態、估計進入不安全狀態機率並在超門檻時介入的路徑；另一條路徑是把工具使用流程以 STPA 等方法變成可執行規格。這些方法的共同點是：把安全控制權從 LLM 的自由推理中抽離出來。

    2025–2026 最值得追蹤的「細粒度指標」與趨勢

    以下指標刻意設計成可直接量測、可跨系統比較、可映射到規劃/反思機制的具體設計旋鈕：

    每決策循環 token 預算（T_cycle）與遵循率：不是只報平均 token，而要報「P95/P99 token」與「超預算比例」，因為受限環境常被 worst-case 佔據。TALE/Budget-forcing/RL 方向的核心價值就在於把推理與反思做成可遵循的 token budget。

    J/token（分 prefill / decode）與 J/success：在 edge 上「每 token 的焦耳」差異可達多倍到數十倍，且受任務影響巨大；真正該追的是「達成任務成功所需的總能量」，這會讓「用更少 token 但更準」的方法（例如 verifier head、PACER）自然浮現。

    反思的邊際收益曲線 ΔSuccess / ΔToken（或 ΔJ）：把反思頻率設計成可調參數（每 N 步、或事件觸發），並用 ROI 曲線決定是否值得做下一次反思。EMNLP 2024 與 AI Agents That Matter 的共同訊息是：不做 ROI 量測的反思架構很容易「花更多、未必更好」。

    Verifier/critic 的成本占比與共享程度：在受限環境，最有前景的是「共享 backbone + KV-cache reuse」的 verifier（幾乎不增加參數搬移與重算），或「單次 revision」的 bounded overhead；追蹤指標可以是「每提升 1% accuracy 所需的額外 token/J」、以及 verifier 是否需要獨立模型載入。

    KV-cache 壓力指標（MB per 1k tokens、swap 次數、或 cache hit/miss）：規劃/反思越長，KV-cache 越大；在行動端管線中這是首要瓶頸，因此任何能在「不重算 prompt」情況下切 mode、或縮短推理 token 的技術，都應該用 cache 壓力指標來量化收益。

    工具呼叫的資訊增益（IG per tool call）與 tool budget：BAVT 的設計提醒：工具呼叫（尤其 search）在成本結構上可能壓倒 token 成本；在離線/間歇連線時更是如此。值得追蹤的是「平均每次工具呼叫帶來的可驗證信息增量」與「被 critic 判定為冗餘而剪枝的比例」。

    資料來源可靠性與不確定性標註

    • 較高可靠性（可重現/同行審視）：USENIX ATC（CLONE）、EMNLP（budget-aware 評估）、Nature/大型期刊與公開基準（例如 Raspberry Pi 量測搭配硬體量測儀器）。
    • 中等可靠性（arXiv/OpenReview，需關注復現）：on-device reasoning pipeline、PACER、BAVT、TokenPowerBench、ML.ENERGY、ProbGuard 等多為可檢視的技術報告/預印本，通常提供充分實驗但仍可能因環境/硬體差異而移植性有限。
    • 較高不確定性（估計/模擬為主或硬體尚未實測）：部分 neuromorphic/spiking transformer 工作承認缺乏實體晶片測量；以及存算一體/類比算子的「orders-of-magnitude」宣稱往往集中在 attention 子模組或特定設定，未必等價於整體 LLM agent 的端到端收益。

    總結而言，2024–2026 最清晰的趨勢是：高度受限環境的 Agentic AI 正在從「更複雜的規劃/反思」轉向「可度量、可控、可路由、可驗證、且與硬體瓶頸共設」的規劃/反思——其中「token budget、J/token、反思 ROI、KV-cache 壓力、工具資訊增益」將比「是否用了 ToT/Reflexion」更能預測該系統能否真正落地。
